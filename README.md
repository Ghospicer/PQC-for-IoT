# PQC-for-IoT: Adaptive Post-Quantum Cryptography Architecture for Critical Industrial IoT

## Overview
The security of Critical Infrastructure (CI) and Industrial Control Systems (ICS) faces an existential threat from the advancement of quantum computing. Long-lived telemetry data in sectors like energy and manufacturing is vulnerable to "Harvest Now, Decrypt Later" attacks, where encrypted traffic is captured today to be decrypted retrospectively once Cryptographically Relevant Quantum Computers (CRQCs) become available.

This project implements a hardware-aware, adaptive Post-Quantum Cryptography (PQC) architecture designed to secure legacy and modern heterogeneous IoT networks against quantum threats without requiring a complete hardware overhaul. 

## System Architecture
The system utilizes a **Dual-Lane Server Architecture** to manage devices with drastically different computational capabilities under a single Unified Gateway.

* **Legacy Field Sensor (ESP32-WROOM-32):** Represents a resource-constrained, low-power monitoring node. To bypass severe memory fragmentation (520 KB RAM), it implements a **"Quantum One-Time Pad"**. Telemetry data is broken into 32-byte chunks, and a fresh ML-KEM-768 encapsulation cycle is performed for every segment, ensuring perfect forward secrecy.
* **Modern Edge Gateway (Raspberry Pi 5):** Represents a high-performance surveillance or data collection node. It utilizes a high-throughput **"Stream Cipher"** lane, leveraging the SHAKE256 Extendable-Output Function (XOF) for real-time bulk encryption.
* **Unified PQC Gateway (Server):** Dynamically applies the appropriate security layer by separating traffic based on the port (Port 65432 for Stream Cipher, Port 65433 for One-Time Pad).

## Cryptographic Standards & Defenses
* **Algorithm:** NIST-standardized FIPS 203 ML-KEM-768 (Kyber).
* **Replay Attack Mitigation:** Implements strict temporal freshness using 64-bit Unix Timestamps embedded within the encrypted payload. The server statefully validates these timestamps, silently dropping any captured-and-resent packets.
* **Man-in-the-Middle (MitM) Prevention:** Utilizes a Pre-Shared Trust (PST) model where the server's public key is hardcoded into the immutable flash memory of the field sensors.
* **Traffic Analysis Defense:** Appends 16 bytes of true random noise generated by the ESP32's Hardware RNG to every payload before encryption, neutralizing Side-Channel and Known-Plaintext attacks.

## Performance & Validation Results
Empirical testing within a simulated industrial testbed confirmed the operational viability of the architecture:

* **Latency (SCADA Compliant):** The ESP32 implementation achieves a highly stable processing latency of **23.01 ms** per chunk. This falls safely within the <100 ms deterministic response time required by real-time industrial control loops (IEC 62443 standards).
* **Entropy Quality:** The random keys generated by the ESP32 achieved a score of **7.999 bits/byte** on the NIST SP 800-22 statistical suite, confirming information-theoretic security suitable for a One-Time Pad.
* **Server Scalability:** Under heavy continuous load, the Raspberry Pi 5 gateway CPU utilization remained strictly under **10%**, validating the efficiency of the hybrid SHAKE256 approach.

## Hardware
* **Sensor Node:** ESP32-WROOM-32 Development Board (ESP-IDF v5.1).
* **Edge Node/Gateway:** Raspberry Pi 5 (4GB RAM) running Debian 12 Bookworm.
